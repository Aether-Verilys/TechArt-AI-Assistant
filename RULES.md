# RULES.md - 项目规则

## 1. 资产命名规范

### 通用规则
- 使用小写字母、数字和下划线
- 避免空格和特殊字符
- 保持一致性

### 命名模式
```
[类型]_[描述]_[变体]_[分辨率/版本]
```

### 示例
- `texture_base_color_rock_01_2k`
- `material_water_waving_01`
- `mesh_character_hero_lod0`

## 2. 着色器代码规范

### 跨引擎兼容性
- 使用通用的数学函数和数据类型
- 避免引擎特定的内置函数
- 提供fallback实现

### 代码结构
```glsl
// 头部：许可证和说明
// 输入：统一的输入结构
// 输出：统一的输出结构
// 函数：模块化、可重用
// 主函数：清晰、简洁
```

### 性能考虑
- 避免分支语句在关键路径
- 合理使用LOD和mipmap
- 考虑移动端限制

## 3. 材质制作规则

### 基本原则
- 保持材质图清晰易读
- 使用有意义的节点命名
- 添加注释说明复杂逻辑

### 性能优化
- 合并相似材质
- 使用材质实例
- 合理使用纹理采样

## 4. 性能预算规则

### 帧时间分配
| 类别 | 目标时间 | 警告阈值 |
|------|----------|----------|
| CPU逻辑 | ≤ 8ms | > 10ms |
| GPU渲染 | ≤ 10ms | > 12ms |
| 内存使用 | ≤ 1.5GB | > 2GB |

### 资产限制
| 资产类型 | 推荐大小 | 最大大小 |
|----------|----------|----------|
| 纹理 | ≤ 2K | 4K |
| 网格 | ≤ 50K tris | 100K tris |
| 材质 | ≤ 50指令 | 100指令 |

## 5. 项目级通用规则

### 安全规则
- 不包含敏感信息（API密钥、密码等）
- 使用环境变量管理配置
- 定期更新依赖包

### 测试规则
- 所有工具必须有单元测试
- 关键路径必须有集成测试
- 保持测试覆盖率 > 80%

### 代码风格
- 遵循PEP 8（Python）或相应语言规范
- 使用有意义的变量名
- 添加必要的注释

### 文档要求
- 所有公共API必须有文档
- 复杂算法必须有解释
- 提供使用示例

## 6. 引擎兼容性规则

### 文件格式
- 使用开放标准格式（如glTF、PNG）
- 避免引擎专有格式
- 提供转换工具

### 功能检测
- 检测目标引擎的支持特性
- 提供降级方案
- 记录兼容性矩阵

## 7. Git工作流

### 分支策略
- `main`: 稳定版本
- `develop`: 开发分支
- `feature/*`: 功能分支
- `hotfix/*`: 紧急修复

### 提交规范
```
类型(范围): 简短描述

详细描述（可选）

关联问题：#123
```

### 类型说明
- `feat`: 新功能
- `fix`: bug修复
- `docs`: 文档更新
- `style`: 代码格式
- `refactor`: 重构
- `test`: 测试相关
- `chore`: 构建/工具

## 9. 质量保证

### 代码审查
- 所有更改必须经过审查
- 至少需要1个批准
- 关注可读性、性能和安全性

### 自动化检查
- 静态代码分析
- 性能基准测试
- 兼容性测试

## 10. 例外处理

### 申请例外
如果需要违反上述规则，必须：
1. 提交例外申请
2. 说明理由和替代方案
3. 获得项目负责人批准

### 例外记录
所有例外必须记录在 `docs/exceptions.md` 中，包括：
- 违反的规则
- 理由
- 批准人
- 有效期
